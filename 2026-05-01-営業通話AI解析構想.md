# 営業通話AI解析構想

> created: 2026-05-01
> tags: #判断 #構想 #Marukin #社内AI #Firebase

## 話題

サーバに通話の音声データが入ったら文字起こし・解析する運用、できるかなって雑談から始まった。スマホで業務通話してるなら、メモの簡略化と「言った言わない問題」防止になるよなと。地方企業の営業電話なんて情報の宝庫だから、ローカルDBに蓄積してけば顧客対応・振り返り・組織知化までいける。

構想段階だからアイディア出し自由でぶん回した。Marukin営業8人想定、会社契約のGoogle Pixel、docomo回線。

## 結論・方針

**Phase 0-1はFirebaseで立ち上げ、Phase 2以降でローカルGPU移行を再評価**するパスが現時点の本命。

構成イメージ:
```
Pixel発着信 → docomoキャリア網で通話録音(NTTドコモビジネス通話録音サービス)
            ↓
         社内NAS(Buffalo TS6000)に音声受信、VPN経由
            ↓
         (Phase 0-1) Firebase Cloud Storageに自動アップロード
            ↓
         Cloud Functions onFinalize で発火
            ↓
         Speech-to-Text + Gemini で文字起こし・要約
            ↓
         Firestore にメタデータ・要約保存
            ↓
         営業マンスマホからPWAで閲覧(本人主権)
```

NTT通話録音サービス(月1,650円〜/回線)は**音声キャプチャ専用**として最小限に使う。後段は自社制御下に置く。

8人運用の年コスト目安:
- Phase 0-1(Firebase): 50-60万/年
- Phase 2-(ローカルGPU移行): 初期70万 + 年10-15万

## 蹴った選択肢

### COTOHA Voice Insight
NTTビジネスの営業通話特化AI解析サービス。一見最強パッケージに見えたけど、**データの可用性に疑問**。生音声データを自由に引き出せるかが不透明で、「ローカルDB蓄積したい」「長期で自社で持ち続けたい」っていう本来の目的と齟齬出る可能性。データ主権譲れないから却下。

### MiiTel / pickupon等のSaaS(050新規番号)
発信時に相手に050番号が表示される。建材業界の長期取引文化だと**知らない番号に出ない・違和感持たれる**リスクがデカい。顧客接点を変えない原則に反するから却下。

### フルAWS / Azure直叩き
できるけど学習コスト高い、Marukin規模で旨味薄い、Firebase一択でいい。

### ローカルGPUいきなり投入
Phase 0-1から70万投資して使われなかったら溶ける。**走りながら改善のスタイル**でやるならクラウドで試して移行判断、が筋。

### Ollama + Open WebUI単独
社内ChatGPT用途には良いけど通話処理パイプには向かない。用途違い。

## 判断軸

- **データ主権**: 顧客通話=個人情報の塊、自社で持ち続けられる構成にしたい
- **メンテコスト・運用継続性**: ベンダーロックイン回避、長期で安く回せる構造
- **ArsMagia知見の流用**: Firebase Auth + Firestore + Cloud Functions + Hosting、ほぼ全部既知
- **段階導入**: Phase 0は最小投資で試す、本番展開は判断後
- **営業マン本人主権**: 監視ツールじゃなく個人秘書ツールとして設計、心理的安全性確保
- **将来のMarukin社内AI基盤**: 通話録音単発じゃなく、社内AI基盤の物理的・組織的足場としても機能させる

## モヤモヤ・揺れ

### NTT通話録音サービスの仕様確認が必須
ここ書面で取らないと全構成崩れる可能性ある:
- 受信サーバの転送プロトコル(SFTP? HTTPS Webhook? 専用クライアント?)
- 音声ファイル形式・サンプリングレート
- **話者分離されてるか(モノラル混合 vs ステレオch分離)** ← クリティカル
- メタデータ(発信者番号・時刻・通話ID等)の形式
- 受信失敗時の再送ポリシー
- オンプレ受信サーバ許容するか・接続要件

法人窓口に問い合わせる時は「COTOHA Voice Insight」じゃなく**通話録音サービス単体の仕様書**って指名する。担当者ガチャ警戒。

### Buffalo NAS と Firebase直送の両立
営業マン手元にデータ欲しい→Firebase直送がラク、はわかる。ただ「ローカルDB蓄積」も諦めたくない。
- 案A: NAS=音声生データ保管、Firebaseには文字起こし結果のみ
- 案B: 両方にコピー(冗長)
- 案C: Firebase主、NASは月次バックアップ
ここ詰めきれてない。

### Phase 2でローカル移行する判断時期
Firebaseのコスト累積がGPU 1台分(70万)を超えるか、社内AI基盤として用途拡張(RAG・OCR・議事録)が始まるか、どっちかが移行トリガー。具体的な数値ライン決めてないので走りながら見る。

### 営業マン心理面の地雷
「通話全部録音される」は監視感強い。設計原則として**本人主権を契約書・運用規程レベルで明文化**って書いたけど、実際導入時に営業さんがどう反応するかは未知数。Phase 1の試験運用で1-2人と話し込んで肌感取りたい。

## 気づき・ツッコミ

### 「ローカルAIサーバ」って24時間動く社内専用Webサービスのこと
いままでローカルAIってなんかぼんやりしてたけど、要するに「**社内専用のSaaSを物理サーバ1台で動かす**」イメージ。Docker Composeで全部定義しておけば移植も再構築も30分。これは腑に落ちた。

### 通話録音は呼び水・本丸はMarukin社内AI基盤
グレードB(70万くらい)のローカルGPUサーバ建てれば、通話処理だけじゃなく社内RAG・議事録自動化・請求書OCR・建材スペック検索・新人教育コンテンツ生成、全部同じ箱で回せる。**稟議書く時は「営業通話分析」より「社内AI基盤(通話分析が第一PoC)」のフレーミングの方が通る**。

### ベテラン営業引退時に資産が蒸発しない
この発想、地味に経営価値ある。Marukinの建材小売は工務店との10年20年スパンの関係前提だから、**通話履歴の組織資産化はベテランの頭の中を残す装置**になる。子供向けIT教育とか地域防災とかとは別軸の、現業価値。

### Firebase直送がラクなのはArsMagia知見ほぼ全部使えるから
Auth、Firestore Security Rules、Functions、Hosting、CLI、デプロイフロー、課金感覚、全部既知。**学習コストゼロで Phase 0-1 立ち上がる**のはデカい。これがなかったら判断違ったかも。ArsMagia無駄じゃなかった。

### Goldilocks原則がここでも効いてる
- 完全クラウド: 自由度低い・コスト累積
- 完全ローカル: 初期投資重い・維持責任全部
- ハイブリッド/段階移行: 中庸でしっくり

自由度高すぎ・制約過剰どっちもNGのいつもの感覚、ちゃんと働いてる。

### 「営業マン手元(忘れ防止メモ用途)」を入れた瞬間にUX設計が変わった
最初は「マネージャ視点の管理ツール」っぽく考えてたけど、本人手元が要件入った瞬間に「**個人秘書アプリ**」に主語が変わった。これ最初に明確化しといて良かった、後から変えると改修地獄。

### NTTサービス名が頻繁に変わってる問題
オフィスリンク、通話録音サービス、COTOHA Voice Insight、Stella AI for Biz、ドコモビジネスへの再編…正式名称・最新仕様は法人窓口に直接確認必須。Web情報は古い可能性常時。
