# 日報PoC 設計詳細 — Chat連携フロー

> created: 2026-05-07
> tags: #判断 #PoC #Marukin #Claude #日報 #Chat #GAS #フロー設計 #実装フェーズ

## 話題

[[2026-04-30-claude日報社内展開PoC設計]] で固めたPoCを実際に2人に配布して走らせ始めた。
通常Project運用で**応答精度の低さ・テンプレ感・要約の浅さ・partnerBrain更新されない問題**が出てきたから、改良案を対話で詰めていった記録。

実機検証は配布2人にしか頼めない制約あり。机上で詰められるとこまで詰めるフェーズ。

## 結論・方針

### 1. 全体フロー(改良後)

```
[モバイル(外勤中)] — 共有前提のフォーマル発信のみ
   ├─ Google Chat 共有スペース → 業務事実共有(訪問録・配達・在庫)
   ├─ Google Chat self-DM → 鮮度確保用の速記メモ(個人raw)
   └─ Pixel通話録音 → recorder.google.com(Phase 0a検証通れば)

[Chat Bot via GAS] — 既に構築済み(入口側は動いてる)
   ↓
[スプシ追記 + Gemini API構造化抽出] — 既存資産
   列構成: 報告日時|日付|報告者|投稿元(新規追加)|報告内容|訪問先|お客様名|作業内容|備考
   ↓
[GAS定時バッチ 18:00]
   投稿元別にMarkdown振り分け生成
   ├─ 投稿元=DM → 各人マイドライブ /daily-personal/YYYY-MM-DD.md
   └─ 投稿元=スペース → 共有Drive /daily-shared/YYYY-MM-DD/各人.md

[帰社後Cowork(デスクトップ)] — 本領発揮
   ├─ Vault素材として読込み
   ├─ 個人raw吐き出し(モヤモヤ・本音・判断揺れ)←←ここが本格raw層
   ├─ 日報整形 → 本人レビュー → 上司向け
   └─ partnerBrain更新提案 → Vault反映
```

### 2. raw吐き出しは「二段階」構造

ここが今回の発見ポイントだった。

- **モバイルrawメモ**(self-DM)= 鮮度確保用の速記
- **Coworkrawメモ**(Vault) = 深掘りの本格吐き出し

これで**音声入力派・移動中思考タイプ**の鮮度が拾えるし、本音は安全な環境(Cowork=完全プライベート)で出せる。
モバイルでは「最初から共有前提」と「速記ラフメモ」だけに絞ることで、自己検閲の二重発生を避けられる。

「外=共有用フォーマル、内=Cowork相棒で本音」の二分割が原則と運用にガチっとハマった。

### 3. raw/共有の分離方式: B案採用(論理分離)

| 案 | 仕組み | 採否 |
|---|---|---|
| **A. スプシ自体を分離** | DM=個人スプシ、スペース=共有スプシ | 採用見送り |
| **B. 単一スプシに「投稿元」列追加** | スプシは混在、出力先で物理分離 | **採用** |

現場の雰囲気的にBで大丈夫、運用した所感での判断。原則よりも実装速度・運用負荷を優先。
ただし `投稿元` 列を入れることで**いつでもAに移行可能な構造**は残しておく。後から本人主権を強化したくなった時の脱出口は確保。

スプシは混在、Markdown出力先は分離、で現場運用シンプル+原則(本人主権)もMarkdown層で担保される設計。

### 4. 通話録音は別チャネル(将来統合)

[[2026-05-01-営業通話AI解析構想]] と接続するけど、Phase 0aは別走り。
- 配布2人は会社支給Pixel 9a持ち、通話録音標準機能日本でも解禁済み
- 顧客同意は問題なし、Marukin的にクリア
- ただしPixel通話録音→Drive自動同期は標準機能ではなさそう(recorder.google.com独自領域問題)
- 実機検証項目をTier1〜3で整理(別途検証手順書を配布2人どちらかに渡す)

通話録音は**rawチャネルへの素材投入経路の一つ**として位置づけ、Chat連携と並走で進める。

### 5. Cowork側精度向上の打ち手(別軸)

通常Project chatの精度問題はChat連携で**実質解消**(Project chat使わない設計に振った)。
ただし配布2人がCoworkに移行できない時のためのProject精度向上案も並行で温める:
- カスタムインストラクションで出力フォーマット強制(YAML+構造化ブロック)
- partnerBrain更新提案ブロックを毎回出力(編集権限ない問題を「人力一手」で解く)
- Project knowledge月次更新ルーティン

これらはChat連携が動けば優先度下がる。Chat連携が本筋。

### 6. 実装ステップ(B案・1〜1.5日想定)

| Step | 内容 | 工数 |
|---|---|---|
| 1 | スプシに `投稿元` 列追加 | 5分 |
| 2 | Bot側で投稿元判別+列書き込み | 30分 |
| 3 | スプシ→Markdown変換スクリプト(振り分け) | 半日 |
| 4 | 出力先フォルダ用意(マイドライブ・共有Drive) | 30分 |
| 5 | 定時トリガー(終業18時バッチ) | 15分 |
| 6 | Cowork側Skill: 当日Markdown読込動作 | 半日 |
| 7 | 動作確認・調整 | 半日 |

## 蹴った選択肢

### Project chat単独で精度を上げ続ける路線
- 構造的に通常Projectは「ファイル直書きできない・Skill使えない・partnerBrain能動更新できない」 の壁がある
- カスタムインストラクション強化やProject knowledge更新で改善はできるが、Coworkとの密度差は埋まらない
- → Project chatは**バックアップ用途**に降格、本筋はChat→Cowork経路に

### Dispatch機能(旧Assign tasks from anywhere)に賭ける
- 改称されてDispatchに、複雑タスク50%成功率で安定不足
- PC常時起動&Coworkアプリ起動が必要
- 安定するまで2〜4ヶ月見込み、PoC期限と合わない
- → 安定したら採用検討、現時点は依存しない

### SSH/リモートデスクトップでモバイルからCowork操作
- Tailscale + Termius等で技術的には可能
- 配布2人にはハードル高すぎ、運用継続性ゼロ

### モバイル雑談用にProject精度向上のみで攻める
- 外勤者の音声入力派には継続使用してもらう想定だったが、構造的限界が大きい
- → モバイル雑談は捨てて、共有事実発信のみに絞る方が清潔

### スプシ物理分離(A案)
- 本人主権原則的には筋通る
- ただし現場の雰囲気的にBで運用問題ない判断
- 工数差(半日)とBotロジック複雑化を考慮、Bで暫定+将来移行可能な構造で十分

### Workspace共有Driveに通話録音を直接自動同期
- Pixel標準機能では成立しない可能性高い(保護ディレクトリ問題、recorder.google.com経由必須)
- → Phase 0aは個人マイドライブ経由、Phase 1で自動転送設計

## 判断軸

- **現場運用性 > 設計原則の完璧主義**(Bで十分、Aは将来オプション)
- **既存資産を壊さない**(GAS Bot+スプシ+Gemini抽出は既に動いてる、ここに継ぎ足し)
- **走りながら改善**(MVPで動かす→運用見ながらPhase 1拡張)
- **他者駆動の自然動機を起点に**(配達リソース管理が現状ダメ→Chat共有スペースで改善+日報PoCに乗せる相乗効果)
- **音声入力派の鮮度確保 vs 本人主権**(self-DM逃げ道で両立)
- **本人レビューゲート必須**(自己検閲を避けるため、生ログは本人だけ・整形済を共有)
- **将来Aに戻せる構造**(投稿元列があればいつでも分離可能)

## モヤモヤ・揺れ

### B案で長期運用した時の心理的安全性
- 「raw部分は管理者が閲覧しない」を運用規程で明文化する予定だが、形式的でしかない
- 配布2人が「本当にraw書ける」と感じるかは未知数
- 1〜2ヶ月運用してから「Aに移行したい」声が出てきたら即対応する、を心構えとして持っておく

### Cowork側Skillが未設計
- `marukin-daily-note.skill` の動作詳細はまだ詰めてない
- Vault読込→partnerBrain参照→raw吐き出し誘導→日報整形→本人レビュー→partnerBrain更新提案、の動線
- 既存obsidian skill流用できるはずだが、PoC2人運用に耐える設計検討要

### 本人レビューゲートのUX
- 削除・赤入れ・承認の動線をどこでやらせる
- Markdownエディタでガリガリ vs Cowork内の対話で「ここ消して」みたいに指示出すか
- IT苦手社員にどっちが続くか未検証

### 上司向け日報のフォーマットと配信経路
- メール? Docs? Chatで上司DM?
- 上司側からの問い返し・コメントの戻しルート
- 「読まれた感」管理(既読通知)

### 過去Markdownの保管設計
- Vault内に日付別フォルダで全部残す前提でいいか
- 容量・検索性・組織知資産化(エンドステート)観点で要詰め
- 月次でアーカイブする運用も検討余地

### 通話録音Phase 0a実機検証はまだ
- 配布2人どちらかに検証手順書を渡す段取り
- 自分がPixel持ってないので机上で詰めきれる範囲が制限される
- C-2(Driveマイドライブにファイル実体ある?)が判定の分岐点

### Geminiの構造化抽出精度の所感
- まだ「試験的に試してる」段階、精度どれくらいかは要観測
- 取引先名辞書をプロンプトに渡せば表記ゆれ吸収できそうだが、未実装
- 誤抽出フィードバックループはまだない

## 気づき・ツッコミ

### Chat Bot+GAS で「取り込み経路」と「フォーマット標準化」が同時解決
これがデカい発見。Botに常駐させてリアクティブ処理にすれば、定時バッチもMarkdown変換も投稿時整形も**1つの仕組みで全部入り**になる。スラッシュコマンド(`/訪問` `/配達` `/在庫`)でテンプレ強制すればフォーマット標準化も同時に効く。**論点1と2を分けて考えてたのが無駄だった**、Bot前提に組めばよかった。

### 「外=共有フォーマル、内=Coworkで本音」の二分割が原則と運用に綺麗にハマった
これ最初から想定してた構造じゃなくて、**精度問題の解決策を探っていたら自然にこの構造に辿り着いた**。前PoC設計の「rawは個人、共有は整形済」原則が実は「**外発信は共有限定、本音発信は内側のみ**」と等価だったことに気づいた。Project chat単独で頑張ろうとしてた発想自体がブレてた。

### 既存資産が想像以上に進んでた
スプシ蓄積機構が動いてるどころか、Gemini API使って自由記述から訪問先・お客様名・作業内容・備考を構造化抽出までしてた。これ知ってればもっと早く設計収束できた。**「現状確認」を最初にすべきだった**、改良案先行で考えすぎた。

### 通話録音→Drive自動同期の標準機能はなかった
Pixelの通話録音はrecorder.google.com独自領域に同期される仕様で、Driveのマイドライブにファイル実体が出てくるとは限らない。前PoC設計時点では「Driveに出てくる」前提で考えてたから、机上設計の前提が一個崩れた。**実機検証を早めにやるべき項目**にランクアップ。

### B案+「投稿元」列で将来Aに移行可能な構造になる
これ地味に重要。**完璧な分離じゃないけど、後から完璧にできる構造**にしておくと、走りながら改善が回しやすい。Goldilocks原則がここでも効いてる。

### スプシ=「データウェアハウス」、Markdown=「日次サマリ」と層を分ける発想
最初は「スプシ→Cowork直読み」を考えたけど、結局Vault側はMarkdown親和性圧倒的に高いから、**変換層を挟む方が合理的**。生データ蓄積層と日次取り込み層の役割分離。過去検索はスプシ(SQL的検索強い)、当日素材はMarkdown(Cowork処理用)、両方の強みが活きる。

### モバイル運用の捨て方
「外勤者は音声入力派だからモバイル捨てられない」が初期前提だったけど、**「モバイル=共有限定、本音は内側」と用途絞れば**Cowork移行できなくても困らない。これ一番の収穫かもしれない。

### Cowork側精度向上の打ち手は「保険」として残す
Chat連携が動けばProject chatは使わなくていいけど、Project精度向上のカスタムインストラクション・Project knowledge更新ルーティンは**配布2人がCowork移行できない時のフォールバック**として温める価値あり。捨てるんじゃなく保留。

### 設計判断スピードが対話で出てくる
今日の議論、わたし(Kaina)単独で詰めてたら数日かかってたはず。**対話で観点投げてもらう→反応する→構造化**のサイクルが回ったから1セッションで骨格固まった。Project chat単独運用が機能してないのは、**Coworkベースじゃないと密度が出ない**話と直結してる。配布2人にも「Coworkあるとここまで密度違うよ」を実感としてどう伝えるかは継続課題。
