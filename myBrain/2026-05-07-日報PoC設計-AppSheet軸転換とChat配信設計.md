# 日報PoC設計 — AppSheet軸転換とChat配信設計

> created: 2026-05-07
> tags: #判断 #PoC #Marukin #Claude #日報 #AppSheet #Vault #Chat配信 #SF #社長相談

## 話題

[[2026-05-07-日報PoC設計-Chat連携フロー]] のChat連携設計で詰まったから、社長と相談。
軸転換の助言をもらって、既存資産(Chat連携・AppSheet・SF・通話録音)の役割を**全部組み替えた**判断ログ。

同日2本目のノートだけど、**1本目から大きく方針転換**してる。Chat連携を主軸に置いてた時点で見えてなかった現場のリアルが、社長相談で一気に表に出てきた。

## 社長判断軸(自身が納得した3点)

1. **Chat能動メモは浸透していない**(社長指摘の通り)
   - 配布2人ともChat投稿が定着しなかった
   - 「書く」アクションが続かない構造問題、自分の弱点と同型
2. **録音もあると強いが、会社規模感を考えるとコストが割に合わない**
   - 営業8人・固定電話メイン・重要顧客限定でも初期投資+月コストが回らない
   - 「あると強い」のは認める、ただし**Marukinサイズには重い**
   - ベテラン引退対応・組織知資産化は別軸で検討
3. **Salesforceが構造化情報の拠点になっていると使いやすそう**
   - SF活用は中長期方針として握る
   - 現状データ薄いため即主軸化は難しいが、**最終的にSF中心の業務情報基盤**を目指す

## 結論・方針

### 1. 戦略転換の流れ

```
[初期] Chat連携軸 → 配布2人で運用、応答精度・テンプレ感・partnerBrain更新で詰まる
   ↓ (社長相談)
[中間] AppSheet軸検討 → 既に浸透中の構造化日次振り返りが見える
   ↓ (Cowork使いやすさ優先で再考)
[現在] Vault Markdown軸 + Cowork対話入力 + Chat+Bot配信
```

軸の根本的な置き換え。Chat連携・AppSheet・SF・通話録音をそれぞれ役割再定義。

### 2. 役割再定義(全体図)

| 入力UI | 役割 | 配布2人の使い方 | 全社運用 |
|---|---|---|---|
| **Cowork対話入力** | 主軸(配布2人) | 終業時の振り返り対話、連絡欄補助 | Phase 1〜段階展開 |
| **AppSheet入力** | 既存浸透層維持 | 配布2人は使わない | 全社員継続(段階移行で縮退) |
| **Chat共有スペース** | リアルタイム業務連携 | 配達・在庫等の即時情報 | 維持 |
| **Chat self-DM / Cowork対話** | 個人raw | Cowork起動時に深掘り | 個人主権前提 |
| **SF** | 構造化情報の拠点(中長期) | 現状はコンテキスト参照のみ | データ蓄積待ち |
| **通話録音** | **当面見送り** | 配布2人運用しない | 将来再検討 |

### 3. Vault Markdown構造(Cowork最適化)

AppSheetの列構成をベースにしつつ、Cowork が読み書きしやすい Markdown に再設計。

```yaml
---
date: 2026-05-07
person: 鳥越
score: 75
condition: 普通
tags: [#日次振り返り, #Marukin]
source: cowork-dialogue
---

# 2026-05-07 日次振り返り

## 総括 / 成果 / トラブル / 目標(明日)

## 上長向け連絡 / 社長向け連絡

---
## 個人メモ・本音(共有しないセクション)
```

frontmatter で構造化、セクション分けで読みやすさ維持、最終セクションを「個人メモ・本音」にして上司向け抽出時は切り捨てる設計。

Vaultフォルダ:
```
/Vault/
   /daily-review/2026-05-07/torigoe.md
   /personal-raw/torigoe/(アクセス権分離)
   /todo/torigoe.md
   /shared/(配達・在庫等)
   /upstream/to-jocho/, to-shacho/
   /partnerBrain-shared/
```

### 4. Chat+Claude対話による配信(構造概要のみ)

詳細は別途。骨格だけ残す。

**スペース構造:**
- `上長宛日報-{社員名}` (本人・上司・Bot)
- `社長宛日報` (管理職・社長・Bot、社長向け連絡記載日のみ配信)
- `部署日報-{部署}` (Phase 2以降)
- 既存「配達リソース管理」継続

**Bot対話補助の方向性:**
- 上司側: 過去類似ケース引き、フィードバックドラフト、月次トレンド
- 本人側: 上司コメント要約、週次FB整理、対応漏れチェック、Todo追加

普通の日報の「書いて送って読む(終わり)」を、**Chat上の双方向対話プラットフォーム**に置き換える設計。詳細は実装フェーズで詰める。

### 5. 全社Vault化ロードマップ(社長と握りたい)

| Phase | 期間 | 内容 |
|---|---|---|
| **0a** | 現在〜2ヶ月 | 配布2人でVault運用確立、AppSheet並走 |
| **0b** | 2〜4ヶ月 | Chat+Bot配信の試験運用 |
| **1** | 4〜8ヶ月 | 希望者拡張(管理職・営業中心) |
| **2** | 8〜12ヶ月 | 部署単位展開、AppSheet↔Vault データ並走→Vault正本化 |
| **3** | 12〜18ヶ月 | 全社Vault化、AppSheetは段階引退 |

各Phase移行のトリガーは**運用所感+データ品質+希望者数**の三点で判断、無理に押さない方針。

### 6. 通話録音の扱い(社長判断で当面見送り)

[[2026-05-01-営業通話AI解析構想]] で詰めてた構想は**Marukin規模ではコスト割に合わない**判断。
ただし「あると強い」価値は否定されてない。条件が揃ったときに再開する設計余地は残す:
- ベテラン引退が現実的な選択肢になった時
- 基幹刷新で固定電話とSFの統合が射程に入った時
- 重要顧客の言った言わない問題で実害が出た時

### 7. SFの位置付け

- **短期**: コンテキスト参照層(○○建設の取引履歴・直近実績を Cowork が引きに行く)
- **中期**: 行動ログ蓄積を全社員に促進(基幹刷新と並走)
- **長期**: 構造化情報の拠点(社長判断軸そのもの)

現状のSF MCPで参照は今すぐできる、書き込みは Phase 1以降。

## 蹴った選択肢

### Chat連携を主軸維持
- 配布2人で投稿浸透しなかった事実、社長も認識
- 普及失敗を引きずると元の木阿弥

### AppSheetをVault側まで真似る
- AppSheetの構造は良いが、**Cowork が一番使いやすい形**を優先する判断
- AppSheetは入力UIとしてのみ既存浸透層に残す

### SF行動ログを即主軸化(社長提案の額面通り採用)
- 現状データ薄い、行動ログは新規追加作業として発生する
- 「能動入力の壁」を別の場所に移すだけになる
- 社長の方向性は正しいが、**現状成立しないことを伝える**判断

### 通話録音Phase 0aの即検証
- Pixelで実機検証手順書まで作りかけてた
- 社長判断のコスト感+モバイルテンポ問題+固定電話インフラゼロベースで凍結

### 配信をDocs/Mailで
- 既存運用と同じ、対話の場にならない
- Chat+Bot対話なら**読み手のフィードバックも構造化される**

### 全社員一斉Vault化
- AppSheet浸透層を壊すリスク
- 段階展開で運用所感を見ながら判断

## 判断軸

- **浸透資産を活かしつつ伸ばす**(AppSheet既存層を否定しない、Vault は段階展開)
- **能動入力の壁を構造で越える**(Cowork対話入力は「答える」だけ、書く能動性が要らない)
- **Cowork中心設計**(Vault Markdown は Cowork が一番使いやすい形)
- **段階移行・走りながら改善**(Phase 0a〜3の見通しは持つが、トリガー次第で柔軟調整)
- **Goldilocks原則**(完全SF軸でも完全Chat軸でも完全自動化でもない、中庸)
- **コストパフォーマンス**(通話録音見送り判断、Marukinサイズに合った投資配分)
- **社長判断を額面通りじゃなく構造で受ける**(SF活用方向性に同意、ただし現状成立しないことは率直に伝える)

## モヤモヤ・揺れ

### 上司側のBot使用習慣化が読めない
- 配布2人の上司がBot対話に乗ってくれるかは未知数
- せっかくの双方向対話設計が「読むだけ」運用になると効果半減
- 最初は配布2人の上司から「Botに慣れてもらう」段階導入が現実

### AppSheet並走→引退のフェーズ移行タイミング
- 全社Vault化のPhase 2-3で AppSheet が縮退する想定だが、判断基準ふんわり
- 「希望者がVault に移ったら AppSheet止める」じゃ遅い、明示的なトリガー設計要

### 通話録音「コスト割に合わない」の試算精度
- 社長判断は妥当だが、**もう一度数字で試算する価値**はある
- ベテラン引退時の暗黙知喪失リスクを定量化したらもしかしたら…
- ただし今の優先度では再検討フェーズじゃない

### SFデータ品質と蓄積ペース未知数
- 「中長期で構造化情報の拠点」が成立するかは現場の入力習慣次第
- 配布2人含めた営業のSF入力習慣化を別軸で進める必要
- 基幹刷新との接続タイミングが鍵

### 全社Vault化の現実性
- 配布2人で成立するかとは別の壁
- AppSheet入力派とVault対話派の比率がどう動くか
- IT苦手中堅層が抵抗ありそう

### 連絡欄(上長向け・社長向け)の効果
- 「人によりけり」の現状を Cowork で揃える設計だが、書き手は揃っても**読み手の上司・社長の活用度**が伴わないと意味ない
- フィードバックループが回る関係性が前提

### Cowork起動できない日の取りこぼし
- 出張・直行直帰・体調不良でPC前にいない日の振り返りどうする
- AppSheet入力で逃げる/モバイルからChat速記/翌朝まとめ、選択肢を残す

## 気づき・ツッコミ

### Chat投稿とAppSheetの浸透差は「人間の認知特性」
- 自由記述(Chat) = 続かない、構造化された問いに答える形(AppSheet) = 続く
- これ自分の「対話で思考が出るタイプ・能動文章化が続かない」と完全に同型
- 個人特性じゃなくて**一般的な人間の認知特性**として運用設計に組み込むべき発見

### Cowork対話入力は「書く」を「答える」に変える
- AppSheetが構造化フォームで「答える」体験だったように、Coworkは対話で「答える」体験
- どちらも能動文章化を回避してる構造
- だから配布2人(と将来の他社員)にも刺さる確度高い

### Vault Markdownは「人間も Cowork も読める中間層」
- AppSheet=機械向き構造化、Chat=人間向き自由記述、その中間がMarkdown
- frontmatter+セクション分けで両方に配慮
- 過去ノートとの相互参照・全文検索もネイティブで強い
- これは Kaina個人 Vault 運用の経験値がそのまま組織展開に効く

### Chat+Bot配信は「読み手の参加」を引き出す
- 普通の日報は一方通行、Chat+Bot対話なら上司の関心事や深掘りが見える
- 上司が部下に直接聞きにくいことをBotに聞ける(過去履歴・類似ケース)
- 認識ズレが減るし、フィードバックも構造化される
- これMarukin社内コミュニケーション自体のアップグレードになる

### Vaultが中央DBになると将来SF統合の入り口になる
- Vault Markdown を構造化して持つと、後で SF とのデータ突合・往復が組める
- 「Vaultで本音と業務を繋いで、SFには業務の構造化部分だけ流す」みたいな整理
- 社長判断軸3「SFが構造化情報の拠点」と将来接続できる伏線

### 通話録音「コスト割に合わない」判断は経営者視点の正論
- 個人としては「あると強い」と思いつつ、Marukinサイズで月数万コストは確かに重い
- 「ROIが見えない投資はしない」は経営判断として正しい
- ただ**ベテラン引退対応のリスク定量化**は別途やった方がいい(現業価値の可視化)

### 社長相談で軸転換できたのは「対話の場」を持てたから
- 自分1人でChat連携の精度問題詰めてたら出口なかった
- 社長視点(経営者・現場の浸透感覚)が入ることで一気に絵が変わった
- 「壁打ち相棒」がClaudeだけじゃダメで、**人間の関係者(社長・配布2人・上司)とも壁打ち回さないと**設計の盲点が消えない
- これは [[partnerBrain]] の関係性レイヤーで握ってる「他者駆動・目的駆動型」と整合

### 「徐々に浸透している既存資産」の発見が鍵だった
- AppSheetが浸透中という事実を最初から知ってればもっと早く設計収束できた
- 改良案先行じゃなく、**現状の浸透資産・失敗資産を最初に棚卸しすべき**
- 次回PoCや別領域への展開でも同じ反省を活かす

### 設計の「二度目の仕切り直し」を許容できる構造が大事
- 同日に2本ノート書いてる(Chat連携フロー → AppSheet軸転換)
- 「**走りながら改善**」が機能してる証拠
- 完璧主義で1回目を作り込まなかったから方針転換コストが小さく済んだ

1. **配布2人にCowork対話入力スキル提供**(0a開始)

9. **SF MCP経由でのコンテキスト参照、配布2人運用で試験**
