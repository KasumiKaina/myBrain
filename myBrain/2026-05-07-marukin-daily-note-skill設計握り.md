# marukin-daily-note skill 設計握り

> created: 2026-05-07
> tags: #判断 #PoC #Marukin #Claude #日報 #skill設計 #partnerBrain #Vault #Cowork

## 話題

[[2026-05-07-日報PoC設計-AppSheet軸転換とChat配信設計]] で固めた方針を受けて、**配布2人向け `marukin-daily-note.skill` の詳細設計**に入るための前提握り。

skill実装に入る前に、未確定の前提が多かったから5論点(A〜E)で先に詰めた判断ログ。

「日報にCoworkを主軸」と言ってるけど、配布2人運用前提で考えると盲点がいくつかあって、特にA(partnerBrainが2人ぶん存在しない)とB(共有Drive運用の心理的安全性)が初期想定から抜けてた。

## 結論・方針

5論点それぞれ確定:

| 論点 | 確定内容 |
|---|---|
| **A. 各人別partnerBrain** | Kainaが手動で**最小限partnerBrain(役職・担当エリア・取引先群)**を2人ずつ作成。本人プロファイルの深化は時間あるとき対話駆動で後追い。 |
| **B. Vault運用形態** | **Workspace共有Drive一本** + Drive Desktopアプリで参照。IT管理者(Kaina)のアクセスは許容、運用規程で明文化。 |
| **C. 対話粒度** | **ハイブリッド**。デフォはフル対話、本人が「短縮で」と指定したら速攻モード。 |
| **D. AppSheet並走** | **配布2人は入力免除**。skillリリース時に社長合意とセットで握る。 |
| **E. 0a→0b移行** | **0a短期化、0b即移行**。Claudeアカウント配布済みで技術ハードル低い、シームレス移行。 |

### Step 1完了 → 次は

- Step 2(同セッション継続): **SKILL.md骨格設計+対話フロー詳細**
- Step 3(同セッション継続): **最小限partnerBrain雛形設計**(skillの入力前提として固める)

## 蹴った選択肢

### A. partnerBrain関連

- **partnerBrain無しで素のCoworkとして動くskill** — skillが出す対話誘導の質が落ちる、配布2人の業務文脈を活かせない。本気でCowork中心設計するなら個別文脈は必須。
- **skill初回起動時に対話で本人プロファイル収集 → 自動partnerBrain生成** — 筋いいけど、初回起動が重くなって継続性リスク。Kainaが先に最小限を作る方が立ち上がり早い。アイデア自体は捨ててなくて、運用中の「気づき→partnerBrain更新提案」として実装する想定で残す。

### B. Vault運用

- **各人マイドライブ + Cowork directory選択** — 物理分離はクリーンだけど、配布2人の Drive Desktop マウント設定 + Cowork directory切り替えがハードル。
- **GitHub Private Repo 2人分** — gitハードル高すぎる、配布2人にgit運用させる前提は無理筋。
- **raw層=マイドライブ / 整形済共有層=共有Drive の二箇所マウント** — 物理分離至上主義としては正論、しかし配布2人とKainaの関係性なら共有Drive一本で成立する判断。「Kainaなら見ても腹くくる」関係性ベースの割り切り。
- **Aに移行可能な構造維持** — フォルダ単位で `personal-raw/` と `daily-review/` を分けておけば、将来必要な時にアクセス権限分離だけで対応可能。今回も「いつでも分離できる構造で書きつつ運用は共有Drive一本」のパターンを採用。

### C. 対話粒度

- **フル対話固定(20分想定)** — 配布2人がそれを「重い」と感じたら継続性ゼロ。
- **3分速攻固定** — 浅すぎてCowork使う意味がなくなる、AppSheetでいい。
- ハイブリッドにして本人選択余地を残すのがGoldilocks。

### D. AppSheet並走

- **Cowork生成Markdown → AppSheet/Sheets自動書き戻し(片方向同期)** — 技術的には可能、ただし二重実装の保守コスト+データ整合性問題。Phase 0aで持つ価値ない。
- **コピペ運用** — 二重入力、絶対続かない。
- AppSheetは段階引退方針なんだから、配布2人だけ先行免除で問題ない判断。

### E. 0a→0b移行

- **0aはVault蓄積のみ(数ヶ月)** — 配信なし期間が長いと「読まれない日報」継続性リスク、書く動機喪失の懸念。
- **上司に手動でMarkdownリンク共有** — 暫定運用としては成立するが、Phase 0bでBot配信に置き換わる時に運用切り替え発生する。
- **Vault directoryを上司にも見せる** — 共有DriveそのままならOKだが、見せ方は0bで設計するからこのフェーズで急ぐ必要なし。
- **Kainaが上司代理で週1介入** — 0a長期化前提なら有効、けど0a短期化したらこの介入も不要。

## 判断軸

- **走りながら改善**(完璧設計より小さく回す)
- **配布2人の継続性最優先**(配布対象が止まったら全部終わる)
- **二重入力禁止**(D判断の根拠)
- **物理分離より関係性ベースで割り切る**(B判断、Marukinサイズの利点)
- **段階移行・トリガー設計**(0a→0b、partnerBrain深化、Vault運用拡大)
- **Goldilocks原則**(C: 自由度と制約のバランス、A: 自動生成の重さと本人駆動の軽さ)
- **Cowork中心設計を貫く**(skill主導で対話完結まで持っていく)

## モヤモヤ・揺れ

### 共有Drive一本の心理的安全性、配布拡大時に崩れる懸念
- 配布2人とKainaは管理者-メンバー関係で「Kainaなら見られても腹くくる」が成立する前提
- 全社展開時に「上司」が見る側に変わると、心理的安全性が一気に崩れる
- フォルダ構造は最初からraw/共有を分けて設計しておかないと、後から分離するの面倒
- → SKILL.md設計で「raw層と整形済層をフォルダレベルで分離した状態でMarkdown出力する」のは死守する

### 最小限partnerBrainの粒度
- 「役職・担当エリア・取引先群」が最小として十分か未検証
- 商材取り扱いの得意不得意・客先の温度感・上司との関係性、業務理解には必要そう
- けど初期に欲張ると作る側(Kaina)の負担で立ち上がり遅れる
- → Step 3 で雛形作る時に「最小」の粒度を再検討、初版は薄くて構わない方針

### ハイブリッド対話の「短縮で」起動方法
- 本人が起動時に「短縮で」と言葉で言うのか、最初に選択肢を提示するのか
- 配布2人がそれを覚えてくれるか問題
- → SKILL.md設計で対話フロー詳細決める時に、selection方式 or 自然言語トリガーを判定

### 0a期間の具体的な長さ
- 「短期」って1週間? 2週間? 1ヶ月?
- skill動作確認 → Bug fix → 配布2人に渡す → 1週間運用感掴む、で**最短2週間**
- 0bの設計詳細はChat+Bot配信構造設計が済んでから、これは並行で詰める
- → Step 4以降で握る

### AppSheet免除の社長合意取得タイミング
- skillリリース前に握る? リリースと同時? リリース後?
- skill開発中の段階で「Phase 0aで2人だけ免除させて」を社長に伝えとく方が清潔
- 全社Vault化ロードマップを社長と握る場と一緒に持っていくのが筋
- → 別軸: 社長相談アジェンダにこれ追加する

### partnerBrain更新提案の受け取り方
- 「対話駆動partnerBrain深化」をskillに仕込むと、配布2人のpartnerBrainが運用中に勝手に育つ設計になる
- けど更新提案を Kaina が承認するフローが要る(配布2人本人の partnerBrain だから本人合意も要るかも)
- → SKILL.md設計で「partnerBrain更新提案セクション」のレビュー動線を詰める

### 「短期 0a」だと skill 安定性が読めないままBot配信に進むリスク
- 0a短期化 = skill運用感の検証期間が短い
- 本来 0a は「skillが配布2人の手で安定動作するか」確認するフェーズだったはず
- 0a→0bシームレス移行は理想だが、skillが詰まった時のリカバリーフェーズ要設計
- → SKILL.md設計で「失敗時のフォールバック動線」を意識する

## 気づき・ツッコミ

### 5論点ぜんぶKainaの思考特性が反映されてる
- A③候補(対話駆動partnerBrain深化) = 対話で思考が出るタイプ
- Cハイブリッド = 自由度高すぎ苦手・制約過剰もNG(Goldilocks)
- B共有Drive一本 = 物理分離至上主義じゃなく、運用継続性重視
- D配布2人免除 = 二重入力嫌い(自分の能動文章化苦手と地続き)
- E0a短期化 = 走りながら改善、書きっぱなし期間最短化

これ、配布2人にもKainaの思考特性が転写される設計になってる。配布対象が他人格(IT苦手中堅とか)になった時、同じ判断軸で設計したら詰まる可能性ある。**他人のpartnerBrain設計の難しさ**が早くも出てきてる。

### 共有Drive一本判断は「Marukinサイズの利点」に立脚してる
- 配布2人 = Kaina直接知ってる、関係性で割り切れる
- これ大企業ならNG判断、Marukinサイズだから成立
- 全社Vault化 Phase 2-3 で配布対象が拡大した瞬間、「物理分離するか運用規程強化するか」の再判断が要る
- 規模感に応じた設計切り替えポイントが Phase 移行のトリガーに組み込めるかも

### 「気づきは partnerBrain 更新提案として出す」を skill に仕込むと A③候補が勝手に成立
- A の③候補(skill初回対話で本人プロファイル収集 → 自動partnerBrain生成)を初期実装で諦めた
- でも運用中に「これpartnerBrainに残しといて」フローを skill が拾えば、結果的に partnerBrain は対話駆動で育つ
- これ Kaina 個人の myBrain 運用と同じ構造
- skillの設計目標として「partnerBrain育成器も兼ねる」を明示しておくと後の拡張がブレない

### 0a短期化は skill 安定性への自信前提
- 「Claudeアカウント配布済みで技術ハードル低い」が0a短期化の論拠
- けど skill そのもののバグ・対話精度問題は別軸
- 0a期間で skill 動作確認 → 修正サイクル回せるか、開発側の体力勝負

### AppSheet並走免除は「skillが代替できる」前提
- skillが安定しない段階で AppSheet 免除すると、配布2人の日報がゼロになる最悪パターン
- 「免除」じゃなく「移行」の文脈で、skill安定確認後に切替えるのが安全
- → 社長合意の文言を「免除」じゃなく「Cowork主軸への移行に伴う AppSheet 入力の段階的縮退」みたいにすると政治的にも筋通る

### Step 1の握りで「実装のための前提条件」が明確になった
- A〜Eを詰めなかったら SKILL.md 設計中に同じ論点が次々出てきて手戻りしてた
- 設計に入る前の「前提握り Step」を1セッション分割いた価値、地味に大きい
- 次のPoC設計でも同じパターン使える(白紙設計に入る前に前提を全部洗い出す)

### partnerBrain雛形設計(Step 3) を SKILL.md より先にやるべきか
- skill が partnerBrain 読み込みするなら、入力前提固めてから対話フロー詰める方が依存方向クリーン
- ただし Step 3 やってる途中で SKILL.md 側の要請が出てきたら戻す必要あり
- → 並行で詰める or partnerBrain 雛形先行、どっちでも対応可能な構造で進める
