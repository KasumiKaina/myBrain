# Claude×日報 社内展開構想 PoC設計

> created: 2026-04-30
> tags: #判断 #PoC #Marukin #Claude #日報 #社内展開 #マスキング #ガバナンス

## 話題

いまわたしがClaude × Obsidianで回してる個人思考ログ運用が刺さってるから、Marukinの社内に「日報に近い役割」で展開できないか検討。
プライバシーやファイル配置に不安あったから、PoC設計を対話で詰めていった記録。

## 結論・方針

### 1. 基本構造:1ソース → 3層出力モデル

```
[対話ログ(自分用・raw)]  ← 共有しない
        ↓ Claude要約
[自分レビュー画面]        ← ここで赤入れ・削除可能
        ↓ 承認
[上司向け日報(整形済)]
        ↓ 上司側で集約・要約
[経営層向けダイジェスト(週次/月次)]
```

- rawログは絶対に自動共有しない(本音・モヤモヤを守る源泉)
- **自分レビューゲート必須**。これがないとセルフ検閲が走って既存日報の二の舞
- 層が上がるほど抽象度上げる

### 2. 普及戦略:書く → 対話する転換

既存日報が普及してないのは「能動的文章化が続かない」から。これ自分の弱点でもあるから、他の社員も同じく刺さらないはず。
→ **「書く日報」じゃなく「Claudeと話してたら日報できてた」**を作るのが本筋。

- AI壁打ち相手として配布(日報目的を前面に出さない)
- 1日の終わりに「今日の話から日報下書き作る?」とClaude側から提案
- 本人承認 → 上司宛へ

### 3. データガバナンス

ゼロから書かない。雛形あるから流用:
- **JDLA「生成AIの利用ガイドライン」第1.1版** ← Word配布、これコピペ改変が中小定番
- **経産省・総務省「AI事業者ガイドライン」第1.1版**(2025年3月) ← 上位枠組み、別添にワークシート
- **東京商工会議所「中小企業のための生成AI活用入門ガイド」** ← 中小特化、社内説明用

JDLA雛形ベースに「目的・適用範囲」「利用可能ツール」「入力禁止情報」「出力物の取り扱い」「インシデント報告」「違反対応」の6項目をMarukin仕様にカスタムすれば半日で叩き台。

### 4. マスキング戦略:対象を絞る

| 対象 | 個人情報? | Marukin運用 |
|---|---|---|
| 法人名(○○建設株式会社) | **対象外** | 素通しOK(機密性は別途NDA観点で管理) |
| 個人事業主・一人親方 | 対象 | コード化 |
| 法人の担当者個人名 | 対象 | 「担当A/B/C」化 |
| 個人連絡先 | 対象 | 削除 |
| 価格・契約条件 | 法的には対象外 | 抽象化(N百万円・利益率X%) |
| 健康・評価情報 | 要配慮 | **入れない** |

→ 法人名コード化は法的にはオーバースペック。**個人名と取引機密だけ守る**で人力コスト8割減る。

### 5. インフラ段階移行

```
Phase 1 (今〜3ヶ月): Cowork個人運用継続 + PoC社員にもCowork
Phase 2 (3〜12ヶ月): Team契約 + Google Workspace連携
Phase 3 (12ヶ月〜): Vertex AI移行 + 自社アプリ
```

PoC段階で「将来API移行を阻害しない設計」をしておくのが肝:
- データ蓄積はClaude内に閉じない(毎回Driveに書き出す)
- マスキング運用を最初から組む(MCP化の予行演習)
- テンプレ・フォーマット統一(構造化処理に乗せる時データ品質揃ってる)
- Cowork固有機能に依存しすぎない

### 6. PoC設計:5枠役割別

横並びにせず、役割別にすると検証ポイントが整理される。

| # | 枠 | 検証ポイント |
|---|---|---|
| 1 | Kaina | 制度設計の妥当性、上限性能 |
| 2 | Kainaの上司 | 2層モデルの読み手体験 |
| 3 | タスク整理困ってる社員 | 困りごとへの効果実感 |
| 4 | 営業/取引先対応系社員 | マスキング運用の負担実態 |
| 5 | IT苦手な中堅 | 展開時のボトルネック層検証 |

→ 5人で5つの仮説検証ができる。

### 7. 保存先設計:共有ドライブ + 個人マイドライブのハイブリッド

| データ種別 | 保存先 | アクセス権 |
|---|---|---|
| 取引先対応表(マスタ) | 共有ドライブ | 5人+上司、編集Kainaのみ |
| ガイドライン | 共有ドライブ | 5人+上司、編集Kainaのみ |
| インシデント報告 | 共有ドライブ | Kainaのみ書き込み |
| 振り返り議事録 | 共有ドライブ | 5人+上司全員 |
| **個人の生会話ログ** | **個人マイドライブ or 個人Vault** | **本人のみ** |
| 上司向け日報(整形済) | 各自→上司の限定Doc | 本人+上司のみ |

**生ログを共有スペースに置かないのが超重要**。共有=自己検閲が走って価値消える。
わたしのObsidian個人運用と同じ構造。生は私物、共有は整形済。

### 8. Skill化で習慣化を仕組み担保

Cowork modeのSkill機能で「marukin-daily-note.skill」を作成:
- トリガー:「日報まとめて」「ノート化」等
- 動作:当日会話の要約 → 個人名チェック → 保存先選択 → 命名規約適用 → Drive保存
- 5人に同じ動作を強制できる(個人差なくなる)

既存obsidian skillを流用すれば数日〜1週間で作れる。

### 9. Salesforce活用で対応表問題ごと解消(条件付き)

MarukinのSF活用で、対応表という概念ごと不要にできる可能性。

| 課題 | 対応表方式 | SF+MCP方式 |
|---|---|---|
| 対象範囲 | 主要10社のみ | 全社カバー |
| メンテナンス | 手動更新 | 不要(SFが正本) |
| 動的追加 | 手動追記+辞書配布 | 即反映 |
| 履歴参照 | 不可 | 過去対応履歴も引ける |
| アクセス制御 | Sheets権限 | SF既存権限体系流用 |

ただし「**MCPで生データ流すなら手入力と同じ**」問題は残る。SOQLでフィールド絞る or カスタムMCPでサーバ側加工が必要。

**最大の収穫:** 建設業特有の「過去のやりとり踏まえた相談」問題、SFで解ける。Activity/Task/Note等を引けば「○○建設さんの過去6ヶ月の対応履歴を踏まえて」が直接できる。

**前提条件(隠れた地雷):** Marukin SFのデータ品質次第。マスタが古い・履歴未入力・カスタム項目乱立だと意味なし。PoC前に品質チェック必須。

## 蹴った選択肢

- **既存日報の延長・改善**
  → 「能動的文章化が普及しない」構造問題に再びぶつかる。延長じゃなく構造転換が必要

- **全社員Obsidian Vault配布**
  → リテラシ・組織権限管理・同期問題で破綻。Obsidianは個人ツール

- **主要10社対応表方式(最終形として)**
  → SF活用が筋通れば全社カバーできるから、対応表は短期つなぎでいい

- **いきなりVertex AI契約**
  → PoC目的は仮説検証、インフラ先行で作り込むと検証結果に合わせて作り直し発生

- **法人名完全コード化**
  → 個情法的にはオーバースペック、人力コスト爆増、生産性下がる

- **モバイル完全対応**
  → IME同期問題が重い。「モバイル=抽象相談専用」と用途絞るのが現実解

- **手入力 vs MCP経由を「同じ」と扱う**
  → Anthropicに送られる事実は同じ、ただしMCPは「フィルタ層挟める」「アクセス制御」「監査ログ」が違う。設計次第で両極端

## 判断軸

- **仮説検証先行・インフラ後追い**(逆だと無駄になりやすい)
- **完璧主義より動かしながら直す**(Goldilocks原則)
- **自己駆動より他者駆動を活用**(困ってる社員の自然動機を起点に)
- **「書く」ではなく「対話する」**(自分の弱点を社員も持つはず、構造で解く)
- **失敗を歓迎する設計**(怒らないインシデント報告がないと萎縮)
- **守るべきものを絞る**(法人名は守らない、個人名と機密は守る)
- **生ログは個人、共有は整形済**(自己検閲を避ける構造分離)

## モヤモヤ・揺れ

- **SFのデータ品質、実際どうなのか未確認**。ここPoC計画の最大ボトルネック候補
- **Skill運用の耐久性**、5人運用に耐えるか・更新配布が回るかPoC内で検証要
- **Phase 2でTeam契約十分回るとVertex AI移行動機が消える「プロ問題」**。それでもいいのかも?
- **iPhone標準IMEがネック**だが、Gboard社内推奨化は政治コスト発生する
- **対象社員がClaude依存になる懸念**。タスク整理苦手な人が「AIなしで動けない」状態化したら本末転倒。月次振り返りで「Claudeなしでもできるようになった部分」も評価項目に入れる予定
- **Gemini for Workspaceとの兼ね合い**。経営から「Claude要らないんじゃ?」が絶対出る。両用使い分け宣言しておかないと荒れる
- **PoC期間中にSF MCP+Skillを開発間に合うか?** 1ヶ月手動運用 → 2ヶ月目導入 → 3ヶ月目検証、の3フェーズに分割するのが現実的かも

## 気づき・ツッコミ

- **「日報書け」じゃなく「話してたら日報できてた」**。心理的負荷ゼロ化が普及戦略の核。これ実は自分のObsidian体験そのもの

- **マスキング対象を絞ると人力コスト8割消える**地味に重要な発見。完璧主義の罠から抜けられるか

- **法人名は個人情報じゃない問題**。建設業界、取引機密としては守るべきだけど個情法対象じゃない。何のために守るかをきちんと分けて考えないとオーバースペック化する

- **建設業の長期関係性ベース判断はAIが原理的に苦手**。30年の付き合いの暗黙知をコードと履歴ログで再現するのは無理筋。Claudeに期待するのは「判断の代替」じゃなく「判断材料の整理・抜け漏れチェック」。**この線引きを最初に握っておかないと過剰期待でガッカリされて普及失敗する**

- **ベテラン社員の頭にしか入ってない暗黙知の文字化、それ自体が別途の事業価値**(属人化解消)。Marukin基幹刷新の中で「やり取りログを構造化して残す設計」を入れ込めるかが分水嶺

- **生ログ共有しない設計が自分のObsidian運用の本質と同じ**。共有=整形=自己検閲=価値消える。これは個人運用も組織展開も同じ構造

- **手入力 vs MCPは「同じに見えて全然違う」**でもあり**「設計が雑なら同じ」**でもある。MCPは「人がブレない」が真価で、Marukin規模なら手入力運用で十分回る可能性も

- **Skill+SF MCPで「対応表」「IME辞書」「マスキング人力」全部消える可能性**、これが一番のゲームチェンジャーかもしれない。SFデータ品質次第だけど

- **個人事業主・一人親方は法人じゃなく個人扱い**。建設業界の取引先構成的に重要な区別。SF側で「個人事業主フラグ」立てる運用が要る

- **Cowork/Team → Vertex AI移行**で「Cowork mode含む便利UI失う」コストは想像以上に重い。自社UI設計者がいないと厳しい。Phase 3に行く判断は慎重に
