### 再燃トリガー = 3つの環境変化の合流 (2024年)

単一要因ではなく、複数の外部環境変化が同時期に重なったことで着手判断に至った:

1. **同種ゲームの市場実例**: 「あるけみすと」プレイで「これは自分でも作れる」と火がついた
2. **公開プラットフォームの整備**: 「あるけみすと」開発者が立ち上げた「ポチゲーポータル」 → 公開先の確保
3. **個人開発の難易度低下**: AIコーディング支援の成熟が見え始めた

これらが同時期に揃ったタイミングを見計らって動いた = 「環境観察→判断→実行」プロセス。

### 2024〜2026の試行錯誤期間

- 公開可能なレベルまで持っていくのが技術的に困難だった時期
- AIコーディング支援の成熟を待ちながら開発を継続
- Firebaseの機能拡充も並行して進行 → スタック選定の追い風に
- 「環境が整うのを待つ」ではなく「**試行錯誤しながら環境成熟と並走した**」期間
- 経営層への翻訳: 技術環境の成熟タイミングを見極めた投資判断

## 動機の三本柱 (確定版・2026-05-13)

ArsMagia着手の動機は3つの構想の合流。

1. **自分の魔法観を作品として体現したい**
   - 魔法を「科学と並ぶ独自の体系」として描きたい(上記コンセプト/魔法観参照)
2. **市場の空白を埋めたい**
   - 「魔法だけで組まれた世界観」を持つゲームが存在しなかった
   - ファンタジーは大抵「剣と魔法」の複合、その**「剣」すらも魔法で再解釈**したい
   - 市場ギャップを明確に認識した上での独自ポジショニング
3. **幼いころから「ゲームを運営してみたい」という構想**
   - 「作る」ではなく**「運営する」側**に立ちたかった
   - 単発の制作物ではなく**持続的な営み**を自分で始めたい志向
   - Kainaの「メンテコスト・運用継続性重視」「百年単位の営みへの敬意」と整合

### 動機の手触り

- **主軸は自己満足ベース**(自分の魔法観を体現したい)
- プレイヤーに遊んでもらいたい気持ちも当然ある(従属的)
- 業務では他者駆動型のKainaが、個人プロジェクトでは自己駆動で長年構想を完遂している点が特徴

### 経営層に語る時の翻訳

- 「長年温めた構想を実行に移した」
- 「思考だけで終わらせず実証する姿勢」
- 「市場ギャップへの問題意識と独自ポジショニング」
- 自己満足部分は無理に経営語に翻訳せず率直に語る方が説得力が出る

## Kainaの魔法観（コア哲学）

- **座右の銘じゃないけど**: 「高度に発達した魔法は奇跡と見分けがつかない」
  - アーサー・C・クラーク「高度に発達した科学技術は魔法と見分けがつかない」の反転
- **魔法 = 科学と同じく独自の理論体系で構築されたスキームの一つ**
  - ファンタジー要素ではなく、もう一つの体系として位置づける
- これは Kainaの「ファクト志向 + 構造観察癖 + 長期営みへの敬意」と整合
  - 神話(根拠ない物語)は「ウソじゃん」判定 / 魔法(構造化された体系)は尊重
  - cf. [[../summary/thinking_style]]「価値観・信念」

### 世界観の含意（技術系統樹の代替分岐）

- **魔法が存在した世界では「電気」が発見されないか、発見されてもかなり遅れる**と考えてる
- 理由: 魔法という代替エネルギー&代替情報伝達手段が先に成立していれば、電気を発見・実用化する社会的駆動力が弱まる
- つまり魔法と電気は同じ役割を担う**代替技術**であり、技術系統樹が分岐するという見立て
- これはSF的にも厚みのある世界観構築の手法

## 経験した障壁・対処・成果 (2026-05-13)

社長報告質問④用に整理。性質の違う2つを並べて対応力の幅を見せる構成。

### ④-1 課金システム審査 + 広告審査 (事務・法務系)

**障壁**
- **Stripe審査**: サイト側整備要件(特商法表示・規約文面等)が厳しく、要件自体も分かりづらかった
- **広告審査(2種類)**: ①広告出稿内容の審査 / ②広告掲載するゲーム側コンテンツの審査
- 個人開発者は専門知識なしに法務・規約系の対応を迫られる構造

**対処**
- Stripe公式窓口へのフォーム問い合わせ
- **最も影響が大きかった = 同じポチゲー運営仲間への相談**
- 一人で抱え込まず、外部リソース(公式窓口・コミュニティ)を使い分けた

**成果**
- 審査通過、安定運営継続中
- 個人開発者コミュニティの横のつながりが課題解決のキーだったと体感

**経営層への翻訳**
- 「外部リソースを使い分ける判断力」を実証
- 個人開発の孤立イメージを覆す「コミュニティ連携での解決」
- 社内にも「困った時の外部相談先」を持つことの重要性へと接続可能

### ④-2 ゲームの満足度・バランス調整 (プロダクト品質系)

**障壁**
- PvPコンテンツがあるため、**スペル間の強弱バランス調整**が継続課題
- 当初は判断軸が定まらず、プレイヤーFBに依存していた

**対処: KPIの自己設計**
- 「スペルの勝利への寄与度」など、運営中に必要なKPIを自分で設計・導入
- 質的判断から定量的判断への移行
- ※③当初目標で「KPI設定なし」と矛盾しない: 当初は質的目標、運営中に必要性を発見してKPIを後付け設計

**成果**
- KPI導入後、調整の精度が向上(自己評価)
- 大きな変更はだいぶ落ち着き、細やかな調整フェーズに移行
- 安定運営継続中

**経営層への翻訳**
- **走りながらKPIを設計する力**: 最初から完璧なKPIは作れない、運営して見えたところで設計するのがリアル
- 「成果への寄与度」の測定設計力 → HR評価・業務改善に応用可能
- Marukinの基幹刷新・HR評価制度・AI推進すべてに通じる話

### ④全体の構造

性質の違う2つで対応力の幅を見せる構成。

| | 性質 | 解決の鍵 |
|---|---|---|
| ④-1 課金審査+広告審査 | 事務・法務系 | **外部リソース活用**(コミュニティ・公式窓口) |
| ④-2 バランス調整 | プロダクト品質系 | **KPIの自己設計**(走りながら改善) |

= 外向き解決力 × 内向き分析力の両方を実証。

## ⑥ 学び (2026-05-13)

### ① AI活用は開発スタイル自体を変える (ε)

- AIをツールとしてではなく「**働き方の前提**」として組み込む実体験
- 実作業時間より対話・思考時間が支配的になる
- ⑧の「AI応答待ちの並行作業」が実証データ
- 経営層への翻訳: Marukin社内AI推進への直接示唆

### ② マネジメント判断と実務の連携の重要性 (ζ)

- ユーザフォーカス起点での意思決定が実務の方向性を決める
- **具体例**: チュートリアル整備(新規プレイヤー向け) vs エンドコンテンツ整備(ヘビーユーザ向け)
- 全員向けにやろうとすると何も刺さらない、フォーカス先を決めてから実務に落とす
- 1人プロジェクトで「判断と実務の両方を回した」からこそ連携の必然性を体感
- 経営層への翻訳: 「新規顧客 vs 既存ロイヤル顧客」のターゲティング、販促・接客・商品開発の「誰向け」起点意思決定 - 事業戦略の根幹
